# เอกสารสรุปและแผนการพัฒนาระบบอนุมัติผลการสอบอัตโนมัติ (Auto-Approve Exam Result)

เอกสารฉบับนี้รวบรวม **(1) สรุปการทำงานของระบบในปัจจุบัน** และ **(2) แผนผังการพัฒนา (Roadmap)** เพื่อขยายระบบให้รองรับรอบสอบพิเศษนิติบุคคลและหน่วยงานภายนอก เพื่อให้สามารถอ้างอิงและอ่านจบได้ในไฟล์เดียว

---

## ส่วนที่ 1: สรุปการทำงานของเมนูกำหนดเงื่อนไขปัจจุบัน (/ExamResultCondition)

### 1.1 จุดประสงค์ของระบบ (Overview)
จากหน้าจอและ Source code เมนู "กำหนดเงื่อนไขการอนุมัติผลการสอบอัตโนมัติ" มีไว้สำหรับ **ตั้งกฎและเงื่อนไข (Rules & Conditions)** เพื่อให้ระบบดำเนินการ **อนุมัติผลการสอบให้กับรอบสอบต่างๆ โดยอัตโนมัติ** แทนที่เจ้าหน้าที่จะต้องมานั่งตรวจสอบและกดอนุมัติด้วยตนเองทีละรอบสอบ ระบบจะทำงานตามช่วงเวลาที่กำหนดและเกณฑ์ของแต่ละศูนย์สอบ ทำให้กระบวนการออกผลสอบเป็นไปอย่างรวดเร็วและเป็นระบบมากขึ้น

### 1.2 การทำงานและฟังก์ชันหลัก (Workflow & Functions)
ฟังก์ชันการทำงานจะแบ่งออกเป็นการกำหนดข้อมูลทั่วไป และการกำหนดเงื่อนไขแยกตามศูนย์สอบ 2 รูปแบบ ดังนี้:

**ก. ข้อมูลทั่วไปของเงื่อนไข**
*   **วันที่เริ่มต้น (Start Date) - วันที่สิ้นสุด (End Date):** สำหรับกำหนดช่วงวันที่ที่กฎ/เงื่อนไขนี้จะมีผลบังคับใช้งาน
*   **รายละเอียด (Description):** บันทึกคำอธิบายว่าทำไมถึงตั้งเงื่อนไขนี้ (Remark)
*   **สถานะการใช้งาน (Status):** สามารถเลือกเปิดใช้งาน (Enable) หรือปิดการใช้งาน (Disable) เงื่อนไขแต่ละตัวได้

**ข. การจัดการเงื่อนไขระดับ "ศูนย์สอบ" (แบ่งเป็น 2 ประเภท)**
*   **ประเภทที่ 1: ศูนย์สอบที่ระบบอนุมัติผลการสอบอัตโนมัติทั้งหมด (Approve All)**
    *   เป็นการระบุรายชื่อ "ศูนย์สอบ" เข้าไปใน List แบบให้ความไว้วางใจสูงสุด
    *   หากมีการจัดการสอบโดยศูนย์สอบที่อยู่ในรายชื่อนี้ ระบบจะ**อนุมัติผลสอบให้ทันที 100%** แบบอัตโนมัติ ไม่ว่าจะเหลือกี่คนหรือผ่านกี่คนก็ตาม
*   **ประเภทที่ 2: ศูนย์สอบที่ระบบอนุมัติผลการสอบอัตโนมัติตาม Option ที่ถูกระบุไว้ (Approve by Condition)**
    *   เป็นการระบุรายชื่อ "ศูนย์สอบ" แบบมีเงื่อนไขผูกมัด ระบบจะอนุมัติผลสอบอัตโนมัติ **ก็ต่อเมื่อ** ภาพรวมผลการสอบของรอบสอบนั้นๆ ตรงตามเกณฑ์ที่ระบุไว้
    *   **เงื่อนไข (Option) ที่ตั้งค่าได้:** ระบุเป็น "รอบสอบที่มีจำนวนผู้สอบผ่านเกิน `[ x ] %` แต่ไม่เกิน `[ y ] %` ของจำนวนผู้สอบทั้งหมด" (หากไม่ตรงเกณฑ์ จะถูกพักไว้ให้เจ้าหน้าที่อนุมัติเอง)

### 1.3 สรุปกลไกการทำงานทางเทคนิค
หน้าจอนี้ทำหน้าที่เป็นหน้าจอ Administrator Configuration สำหรับควบคุมพฤติกรรมของระบบ Auto-Approve โดยทำหน้าที่บันทึก "กฎ" (Rule) ลงใน Database ตาราง `MT_T_EXAM_RESULT_CONDITION` จากนั้นระบบ Background Process (Batch Job: `P_Exam_Result_Transfer_Process`) จะมาอ่านค่าไปประมวลผลหลังจากการสอบเสร็จสิ้น **(ซึ่งปัจจุบันรองรับเฉพาะการดึงไปใช้กับรอบสอบบุคคลธรรมดาเท่านั้น)**

## ส่วนที่ 2: Implementation Roadmap การขยายระบบให้รองรับรอบสอบนิติบุคคลและหน่วยงานภายนอก (v1.0)

หากต้องการขยายให้ระบบ Auto-Approve ครอบคลุมรอบสอบประเภท "นิติบุคคล" (Corporate) และ "หน่วยงานภายนอก" นี่คือแผนผังสิ่งที่ต้องดำเนินการครับ

### 2.1 สิ่งที่ต้องดำเนินการ (Action Items)

**1) การปรับปรุง Database Schema และ UI (Configuration)**
ปัจจุบันตารางและหน้าจอกำหนดแค่ช่วงวันที่และ "ศูนย์สอบ" (`EXAM_CENTER_ID`) โดยไม่ได้แยกว่าเป็นรอบสอบประเภทใด
*   **เพิ่มประเภทเงื่อนไข (Exam Round Type):** ต้องเพิ่ม Field/Checkbox บนหน้าจอเพื่อให้ Admin เลือกว่า เงื่อนไขนี้ใช้กับ 1. รอบบุคคลธรรมดา, 2. นิติบุคคล หรือ 3. หน่วยงานภายนอก
*   **ปรับแก้การผูกข้อมูลสถานที่สอบ:** การสอบรอบนิติบุคคล/หน่วยงานภายนอก อาจจัดสอบแบบ "นอกสถานที่" (On-site ที่บริษัท) ซึ่งอาจไม่มีข้อมูลรหัส `EXAM_CENTER_ID` ตายตัว ต้องเพิ่มการตั้งค่าเงื่อนไขอิงจาก **"รายชื่อบริษัท/รหัสนิติบุคคล" (CORP_ID)**

**2) การปรับปรุง Background Process (Batch Job / Transfer Process)**
*   **สร้าง/อัปเดต Batch Job นิติบุคคล:** ค้นหา Batch Job ที่นำเข้าคะแนนรอบสอบนิติบุคคล (เช่น `C_Exam_Result_Transfer_Process`) แล้วนำ Logic การคำนวณเดียวกันนี้ไปฝัง
*   **คัดกรองจาก Testing No ใหม่:** เปลี่ยนเงื่อนไขที่ดักเฉพาะรอบสอบบุคคลธรรมดา `if (ROUNDType == "1")` ให้รองรับรอบนิติบุคคลและหน่วยงานภายนอก
*   **เปลี่ยนตารางเป้าหมาย:** 
    *   จาก Update ตารางรอบสอบบุคคลธรรมดา (`MT_T_P_EXAM_ROUND`) ➡️ เป็นตารางรอบสอบนิติบุคคล (`MT_T_C_EXAM_ROUND`)
    *   จาก Update คำร้องผู้สมัครบุคคลธรรมดา (`IIQE_T_P_EXAM_REQ`) ➡️ เป็นคำร้องผู้สมัครนิติบุคคล (`IIQE_T_C_EXAM_REQ`)

**3) การประมวลผลและการบันทึก Audit**
*   เขียน Logic คำนวณเปอร์เซ็นต์ผู้สอบผ่าน สำหรับนิติบุคคลให้ถูกต้อง และอัปเดตสถานะ (Flow Status) ให้เป็น "อนุมัติอัตโนมัติ" ตาม Flow นิติบุคคล

### 2.2 ระยะเวลาที่ใช้ในการพัฒนาโดยประมาณ (Estimated Timeline)

**รวมระยะเวลาประมาณ 7 - 12 วันทำการ (Working Days)** ขึ้นอยู่กับความซับซ้อนของโครงสร้างรอบนิติบุคคลปัจจุบัน:

| ระยะการพัฒนา (Phase) | รายละเอียดงาน (Tasks) | เวลาที่ใช้ |
| :--- | :--- | :--- |
| **Phase 1: Analysis & Database Check** | ตรวจสอบ Data Flow ของคะแนนรอบสอบนิติบุคคล และออกแบบตารางและ Field ใหม่เพื่อรองรับเงื่อนไขนิติบุคคล | 1 - 2 วัน |
| **Phase 2: UI & Backend API** | ปรับปรุงหน้าจอ Administrator (`/ExamResultCondition`) ให้มีตัวเลือกรองรับนิติบุคคลและหน่วยงานภายนอก | 2 - 3 วัน |
| **Phase 3: Core Batch Process Logic** | นำ Logic Auto-Approve ไปฝังในระบบนำเข้าคะแนนรอบสอบนิติบุคคล และปรับเปลี่ยน Database Query | 3 - 5 วัน |
| **Phase 4: SIT, Testing & UAT** | จำลองระบบ (Mock) การนำเข้าผลคะแนนรอบสอบปกติ, รอบนิติบุคคล และตรวจสอบความถูกต้อง (รวม Regression Test) | 1 - 2 วัน |

*หมายเหตุ: หากพบว่าโครงสร้างคะแนนรอบนิติบุคคลมีความซับซ้อนมาก เวลาการพัฒนาใน Phase 3 อาจต้องเพิ่มขึ้น*

## ส่วนที่ 3: คำถามเพื่อรวบรวม Requirement เพิ่มเติมจาก Owner

เพื่อให้นำไปประเมินกรอบการทำงานในเชิง Technical ได้แม่นยำขึ้น ควรนำคำถามเหล่านี้ไปสอบถาม Product Owner / System Owner ก่อนเริ่มพัฒนา:

### หมวดที่ 1: การกำหนดเงื่อนไข (Condition Setup & UI)
1. **การระบุเป้าหมาย (Target Identification):**
   * *คำถาม:* ในรอบสอบปกติ ผูกเงื่อนไขกับ "รหัสศูนย์สอบ" แต่สำหรับรอบสอบนิติบุคคลและหน่วยงานภายนอก (ซึ่งอาจจัดสอบนอกสถานที่) เราจะให้ระบบจำแนกและผูกเงื่อนไขด้วยอะไร? (เช่น รหัสบริษัท/Corp ID, รหัสหน่วยงาน, หรือยังมีรหัสศูนย์สอบในระบบอยู่)
2. **ความต้องการแยกหน้าจอ (UI Design):**
   * *คำถาม:* ต้องการให้ปรับเพิ่มตัวเลือกในหน้าจอเดิม (`/ExamResultCondition`) ให้รองรับทั้ง 3 ประเภท (ปกติ, นิติบุคคล, หน่วยงานภายนอก) เลยหรือไม่? หรืออยากให้แยกเมนูตั้งค่าสำหรับนิติบุคคลออกมาต่างหากเพื่อป้องกันความสับสน?
3. **ความแตกต่างของเกณฑ์ (Criteria Differences):**
   * *คำถาม:* เงื่อนไขของนิติบุคคล/หน่วยงานภายนอก จะเหมือนกับรอบปกติ 100% หรือไม่? เช่น มีทั้งแบบ "อนุมัติให้เลย (Approve All)" และแบบ "พิจารณาจากเปอร์เซ็นต์คนสอบผ่าน" 
4. **เคลียร์เงื่อนไขเก่าที่ไม่ได้ใช้:**
   * *คำถาม:* ปัจจุบันในโค้ดมีการซ่อนเงื่อนไขประเภท *"รอบสอบที่มีผู้ผ่านไม่เกิน X%"* เอาไว้ ในอนาคตมีแผนจะใช้เงื่อนไขนี้อีกหรือไม่ หรือให้ตัดออกจากระบบถาวรเลย?

### หมวดที่ 2: กระบวนการและข้อจำกัดทางเทคนิค (Process & Technical Integration)
5. **กระบวนการนำเข้าคะแนน (Data Flow):**
   * *คำถาม:* ปัจจุบันผลคะแนนสอบรอบนิติบุคคลและหน่วยงานภายนอก ถูกส่งเข้ามาในระบบด้วยช่องทางไหน? (เช่น มี Batch Job ดึงข้อมูล, อัปโหลด Excel โดยเจ้าหน้าที่, หรือวิ่งผ่าน API) *เพื่อให้ Developer นำ Logic Auto-approve ไปดักไว้ให้ถูกจุด*
6. **การจัดการกรณีที่ไม่ผ่านเกณฑ์ (Fallback Process):**
   * *คำถาม:* ถ้ารอบสอบนิติบุคคลนั้นผลคะแนน "ไม่อยู่ในเกณฑ์" ที่จะ Auto-Approve ระบบจะต้องเปลี่ยนสถานะไปรอให้ใครมากดอนุมัติ Manual แบบเดิม? (ใช้กลุ่ม User/Role เดียวกับรอบปกติหรือไม่)
7. **เกณฑ์คะแนนสอบผ่าน (Score Passing Master):**
   * *คำถาม:* เปอร์เซ็นต์การสอบผ่านที่จะเอามาใช้คำนวณ ของนิติบุคคลและหน่วยงานภายนอก อิงจาก Master Data ของตารางคะแนนสอบผ่านชุดเดียวกับบุคคลธรรมดาเลยใช่หรือไม่? หรือมีเกณฑ์คิดเปอร์เซ็นต์พิเศษแยกต่างหาก?
